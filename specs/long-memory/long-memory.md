# Electron 长期记忆方案设计

## 1. 方案可行性分析

### 1.1 当前架构

LobsterAI 使用 `sql.js` (SQLite 的 WebAssembly 实现)，不支持加载原生扩展。

### 1.2 关键限制与突破

| 限制/突破 | 说明 |
|-----------|------|
| **sql.js 不支持扩展** | WASM 版 SQLite 无法加载原生扩展 |
| **Node.js 23.5.0+ 突破** | 内置 `node:sqlite` 模块，支持 `allowExtension: true` |
| **sqlite-vec NPM 包** | 可直接加载向量扩展 |

### 1.3 三层降级策略

```
┌─────────────────────────────────────────────────────────────────────┐
│                      三层降级策略                                    │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  Layer 1: sqlite-vec (原生向量)                                     │
│  ├── 条件: sqlite-vec 扩展可加载                                    │
│  ├── 性能: 最优                                                     │
│  └── 降级触发: 加载失败、CPU 架构不支持、系统限制                     │
│              │                                                      │
│              ▼                                                      │
│  Layer 2: 纯 JS 向量 (余弦相似度)                                    │
│  ├── 条件: Embedding 模型已配置                                     │
│  ├── 性能: 中等                                                     │
│  └── 降级触发: 未配置 Embedding API、API 不可用                      │
│              │                                                      │
│              ▼                                                      │
│  Layer 3: FTS5 关键词 (无向量)                                       │
│  ├── 条件: 无 (始终可用)                                            │
│  ├── 性能: 良好 (语义理解较弱)                                      │
│  └── 特点: 纯文本匹配，BM25 排序                                    │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**降级场景**：

| 场景 | 降级到 | 原因 |
|------|--------|------|
| musl libc (Alpine Linux) | JS 向量/关键词 | sqlite-vec 预编译不兼容 |
| 老旧 CPU 架构 | JS 向量/关键词 | 预编译二进制不支持 |
| Windows ARM | JS 向量/关键词 | 可能无对应预编译版本 |
| 未配置 Embedding | 关键词 | 无向量化能力 |
| Embedding API 不可用 | 关键词 | 网络或配额问题 |
| 加载超时 (>30s) | JS 向量 | 扩展加载异常 |

```
│  ┌─────────────────┐  ┌─────────────────────────────────────────┐      │

│  │   人类可编辑     │  │   └── ...                                │      │
│  │   定时提炼生成   │  │   仅追加，保留 N 天                      │      │
│  └────────┬────────┘  └───────────────┬─────────────────────────┘      │
│           │                           │                                  │

│  │  │ (核心表)│ │(FTS5索引)│ │(向量索引)  │ │ (向量缓存)    │    │    │
│  │  └─────────┘ └──────────┘ └────────────┘ └───────────────┘    │    │
│  └─────────────────────────────────────────────────────────────────┘    │
│                          │                                              │
│                          ▼                                              │
│  ┌─────────────────────────────────────────────────────────────────┐    │
│  │                     向量嵌入服务                                 │    │
│  │              (OpenAI / Ollama / 自定义)                          │    │
│  └─────────────────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2.2 设计原则

| 原则 | 说明 |
|------|------|
| **文件为主** | Markdown 文件是 Source of Truth，SQLite 仅作为索引 |
| **单向同步** | Markdown → SQLite，不反向写入 Markdown |
| **人类可编辑** | MEMORY.md 支持用户直接编辑 |
| **非侵入式** | 通过中间件实现，不修改 Agent 代码 |

## 3. 文件存储设计

### 3.1 文件布局

```
workspace/
├── MEMORY.md                 # 核心长期记忆 (精心整理，人类可编辑)
├── memory/                   # 每日记忆目录
│   ├── 2025-03-01.md         # 每日日志 (仅追加)
│   ├── 2025-03-02.md
│   └── ...
└── .memory/
    └── index.sqlite          # 向量索引数据库
```

### 3.2 MEMORY.md 格式规范

```markdown
# 长期记忆

## 用户档案
- 名字: 张三
- 职业: 软件工程师

## 偏好
- 编程语言: TypeScript, Rust
- 编辑器: VS Code

## 项目相关
- 当前项目: LobsterAI

## 重要决策
- 2025-03-01: 决定使用三层降级方案
```

### 3.3 每日记忆文件格式规范

**文件命名**: `YYYY-MM-DD.md` (如 `2025-03-02.md`)

**格式示例**:

```markdown
# 2025-03-02

---

### 14:30 会话
- 用户喜欢 TypeScript 和 Rust
- 当前项目是 LobsterAI

---

### 15:00 会话
- 用户偏好 VS Code 编辑器
- 工作时间 9:00 - 18:00

---

### 16:45 会话
- 用户决定使用三层降级策略
- sqlite-vec 作为首选方案
```

**格式要点**:
- 文件顶部使用 `#` 一级标题标记日期
- 日期标题后紧跟 `---` 分隔符
- 每个追加的记忆使用 `###` 三级标题标记时间和主题
- 每个条目之间使用 `---` 分隔符分隔
- 每个条目内部使用列表项记录具体内容
- 时间戳直接体现在标题中，无需额外标记

**追加逻辑**:
- 新记忆追加到文件末尾
- 如果是当天第一条，先创建文件并写入日期标题 + `---`
- 后续追加时，在文件末尾添加 `---` 分隔符 + 新的 `###` 标题块

### 4.3 同步状态管理

**状态字段** (存储在 memory_meta 表):

```
┌─────────────────────────────────────────────────────────────────┐
│                     同步状态字段                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  dirty: boolean                                                 │
│  ├── true: 有待同步的文件变更                                    │
│  └── false: 索引与文件同步                                      │
│                                                                 │
│  syncing: boolean                                               │
│  ├── true: 同步进行中                                           │
│  └── false: 无同步任务                                          │
│                                                                 │
│  last_sync_time: integer                                        │
│  └── 最后同步完成的时间戳                                        │
│                                                                 │
│  sync_version: integer                                          │
│  └── 同步版本号 (每次同步 +1，用于并发控制)                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**Agent 会话启动时的同步逻辑**:

```
async function ensureMemoryReadyForSession(): Promise<void> {
  // 1. 检查 dirty 状态
  const { dirty, syncing, lastSyncTime } = getSyncState();

  if (!dirty) {
    // 索引是最新的，直接返回
    return;
  }

  // 2. 检查是否正在同步
  if (syncing) {
    // 等待同步完成 (最多 5 秒)
    await waitForSyncComplete({ timeout: 5000 });
    return;
  }

  // 3. 触发增量同步
  // 只检查最近变化的文件，不做全量扫描
  const changedFiles = await detectRecentChanges({ since: lastSyncTime });

  if (changedFiles.length > 0) {
    await syncFiles(changedFiles);
  }

  // 4. 更新状态
  setSyncState({ dirty: false, syncing: false, lastSyncTime: Date.now() });
}
```

**性能优化**:

| 优化点 | 策略 |
|--------|------|
| 避免重复同步 | syncing 标志 + 版本号检查 |
| 快速路径 | dirty=false 时直接跳过 |
| 超时保护 | 等待同步最多 5 秒，超时后使用现有索引 |
| 增量检测 | 只检查 lastSyncTime 后变化的文件 |

### 4.2 文件 Hash 校验机制

**目的**: 检测文件内容是否发生变化，避免不必要的同步操作。

**Hash 存储位置**: SQLite 数据库的 `memory_meta` 表

**校验流程**:

```
┌─────────────────────────────────────────────────────────────────┐
│                   客户端启动时的 Hash 校验                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 读取所有记忆文件列表                                         │
│     ├── MEMORY.md                                               │
│     └── memory/*.md                                             │
│                                                                 │
│  2. 对每个文件计算 SHA256 Hash                                   │
│     └── hash = SHA256(文件内容)                                 │
│                                                                 │
│  3. 从数据库读取上次记录的 Hash                                  │
│     └── SELECT value FROM memory_meta WHERE key = 'file_hashes' │
│                                                                 │
│  4. 比对 Hash                                                   │
│     ├── Hash 相同 → 跳过该文件                                  │
│     ├── Hash 不同 → 标记为需要同步                              │
│     └── 新文件 → 标记为需要同步                                 │
│                                                                 │
│  5. 对需要同步的文件执行增量同步                                 │
│                                                                 │
│  6. 更新数据库中的 Hash 记录                                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**Hash 比对逻辑**:

| 场景 | 文件 Hash | 数据库 Hash | 操作 |
|------|-----------|-------------|------|
| 文件未变化 | A | A | 跳过 |
| 文件已修改 | B | A | 增量同步 |
| 新文件 | C | 无记录 | 全量同步 |
| 文件已删除 | 无 | D | 删除对应记录 |

### 4.3 文件变更监控机制

**监控范围**:
- `MEMORY.md` 文件
- `memory/` 目录下的所有 `.md` 文件

**监控实现方式**:
- 使用 `chokidar` 库进行文件系统监听
- 防抖处理：变更后等待 1.5 秒再触发同步，避免频繁操作

```
│                   文件变更事件处理                               │

│                                                                 │
│  事件: add (文件新增)                                            │
│  ├── 解析 Markdown 文件                                          │
│  ├── 分块并计算 Hash                                             │
│  ├── 向量化 (如果启用)                                           │
│  └── 写入 SQLite                                                 │
│                                                                 │
│  事件: change (文件修改)                                         │
│  ├── 重新解析文件                                                │
│  ├── 比对新旧 Hash，找出变化的 chunks                            │
│  ├── 删除已移除的 chunks                                         │

│  └── 重新向量化变化的 chunks                                     │

```

## 5. 记忆提取策略

### 5.1 核心问题: 大对话的记忆提取

Agent 对话可能非常长 (数百轮、单条消息数千字符)。原始设计中滑动窗口固定 10 条消息，超出窗口的消息直接丢弃，导致**长对话中早期的重要记忆丢失**。

**问题分解**:

| 问题 | 说明 |
|------|------|
| 消息量大 | 对话可达数百轮，远超内存窗口 |
| 单条消息大 | Agent 回复可达数千字符，含代码、分析等 |
| 上下文受限 | LLM 提取的 token 预算有限 (500-2000 token) |
| 队列易失 | 内存中的提取队列进程重启后丢失 |
| 窗口丢弃 | 超出滑动窗口的消息无法回溯提取 |

**解决思路**: 会话转录持久化 + 分段提取 + 跨段去重。

参考 OpenClaw 的做法:
- OpenClaw 将会话完整保存为 JSONL 文件，事后异步处理
- 使用 256 token/chunk、32 token overlap 的分块策略
- 批量嵌入时按 8000 token/batch 控制 API 调用量

### 5.2 会话转录持久化 (Session Transcript)

**核心思想**: 对话消息实时追加到本地文件 (JSONL 格式)，作为记忆提取的完整数据源。

#### 5.2.1 文件布局

```
workspace/
├── MEMORY.md
├── memory/
│   └── 2025-03-04.md
└── .memory/
    ├── index.sqlite
    └── transcripts/                   # 会话转录目录 (新增)
        ├── session-abc123.jsonl       # 会话 A 的转录
        ├── session-def456.jsonl       # 会话 B 的转录
        └── ...
```

#### 5.2.2 JSONL 格式

每条消息追加一行 JSON:

```jsonl
{"ts":1709542800000,"role":"user","content":"我叫张三，是一名 Rust 开发者","msgId":"msg-001"}
{"ts":1709542805000,"role":"assistant","content":"你好张三！很高兴认识你...","msgId":"msg-002"}
{"ts":1709542900000,"role":"user","content":"帮我写一个 HTTP 服务器...","msgId":"msg-003"}
{"ts":1709542950000,"role":"assistant","content":"好的，这是一个基于 axum 的...（很长的代码回复）","msgId":"msg-004"}
```

**字段说明**:

| 字段 | 类型 | 说明 |
|------|------|------|
| ts | number | Unix 毫秒时间戳 |
| role | string | user / assistant |
| content | string | 消息文本内容 (仅保留 text，过滤 tool_use/tool_result) |
| msgId | string | 消息唯一 ID |

**写入时机**: 每条消息到达时立即追加 (append-only)，不阻塞对话。

#### 5.2.3 写入逻辑

```
┌─────────────────────────────────────────────────────────────────┐
│                   会话转录写入流程                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Agent 消息事件到达                                              │
│       │                                                         │
│       ├── 1. 过滤: 仅保留 user/assistant 的文本消息             │
│       │    └── 跳过: tool_use, tool_result, system 消息         │
│       │                                                         │
│       ├── 2. 内容清洗                                            │
│       │    ├── 多行内容中的代码块: 保留但标记 [CODE_BLOCK]       │
│       │    ├── 超长 tool_result: 截断到 500 字符 + "..."        │
│       │    └── 脱敏: 移除 API Key、Token 等敏感信息              │
│       │                                                         │
│       ├── 3. 构建 JSONL 行                                       │
│       │    └── JSON.stringify({ ts, role, content, msgId })     │
│       │                                                         │
│       └── 4. 追加写入文件 (fs.appendFile)                        │
│            └── 文件路径: .memory/transcripts/{sessionId}.jsonl  │
│                                                                 │
│  注意: 追加写入是原子操作 (单行)，无需加锁                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 5.3 分段提取策略 (Segmented Extraction)

**核心思想**: 将长对话按段 (Segment) 处理，每段独立提取记忆，跨段去重。

#### 5.3.1 分段策略

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         分段策略设计                                         │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  对话消息流:                                                                │
│  ┌────┬────┬────┬────┬────┬────┬────┬────┬────┬────┬────┬────┐            │
│  │ M1 │ M2 │ M3 │ M4 │ M5 │ M6 │ M7 │ M8 │ M9 │M10 │M11 │M12 │            │
│  └────┴────┴────┴────┴────┴────┴────┴────┴────┴────┴────┴────┘            │
│                                                                             │
│  分段方式 (segmentSize=5, overlap=2):                                       │
│                                                                             │
│  Segment 1: ┌────┬────┬────┬────┬────┐                                     │
│             │ M1 │ M2 │ M3 │ M4 │ M5 │                                     │
│             └────┴────┴────┴────┴────┘                                     │
│                                                                             │
│  Segment 2:           ┌────┬────┬────┬────┬────┐                           │
│                       │ M4 │ M5 │ M6 │ M7 │ M8 │                           │
│                       └────┴────┴────┴────┴────┘                           │
│                                                                             │
│  Segment 3:                     ┌────┬────┬────┬────┬────┐                 │
│                                 │ M7 │ M8 │ M9 │M10 │M11 │                 │
│                                 └────┴────┴────┴────┴────┘                 │
│                                                                             │
│  Segment 4:                               ┌────┬────┬────┐                 │
│                                           │M10 │M11 │M12 │ (尾部不足时)    │
│                                           └────┴────┴────┘                 │
│                                                                             │
│  重叠区域 (overlap=2):                                                      │
│  - 确保跨段边界的上下文不丢失                                               │
│  - 重叠区域产生的重复记忆通过 fingerprint 去重                              │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

**分段参数**:

| 参数 | 默认值 | 说明 |
|------|--------|------|
| segmentSize | 5 | 每段包含的消息数 (user+assistant 各算一条) |
| segmentOverlap | 2 | 相邻段的重叠消息数 |
| maxSegmentTokens | 4000 | 单段最大 token 数 (超过则动态缩短) |
| maxContentPerMessage | 1500 | 单条消息最大字符数 (超过则摘要) |

#### 5.3.2 Token 限制处理

**Token 估算方法**:
- 中文字符: ~1.5 tokens/字
- 英文单词: ~1.3 tokens/词
- 其他字符: ~0.25 tokens/字符
- 每条消息的 role 前缀: ~2 tokens

**分段时的 Token 检查流程**:

```
┌─────────────────────────────────────────────────────────────────┐
│                    Token 限制处理流程                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  构建分段 (最多 segmentSize 条消息)                              │
│       │                                                         │
│       ▼                                                         │
│  估算分段总 token 数                                             │
│       │                                                         │
│       ├── token <= maxSegmentTokens (4000)?                     │
│       │   └── YES → 直接使用该分段                               │
│       │                                                         │
│       └── NO → 逐条移除消息直到满足限制                          │
│            │                                                    │
│            ├── 剩余 > 1 条消息? → 移除最后一条，重新检查          │
│            │                                                    │
│            └── 剩余 = 1 条消息但仍超限?                           │
│                 └── 截断该消息内容到 token 限制                   │
│                     └── 保留头部 45% + 尾部 45%                  │
│                                                                 │
│  注意: 即使单条消息被截断，也能提取到有效记忆                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 5.3.3 单条大消息的处理

Agent 回复可能包含大段代码、分析报告等。需要对超长单条消息做预处理:

```
┌─────────────────────────────────────────────────────────────────┐
│               单条大消息预处理策略                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  消息内容                                                       │
│       │                                                         │
│       ▼                                                         │
│  长度 <= maxContentPerMessage (1500 字符)?                       │
│       │                                                         │
│       ├── YES → 直接使用                                        │
│       │                                                         │
│       └── NO → 内容分类处理                                     │
│            │                                                    │
│            ├── 纯代码块 (```...```)                              │
│            │   └── 替换为: "[代码: {语言}, {行数}行]"            │
│            │       保留代码前后的说明性文字                      │
│            │                                                    │
│            ├── 混合内容 (文字 + 代码)                             │
│            │   ├── 保留所有非代码文字                            │
│            │   └── 代码块替换为摘要标记                          │
│            │                                                    │
│            └── 纯文字但过长                                     │
│                ├── 保留前 500 字符                               │
│                ├── 保留后 500 字符                               │
│                └── 中间替换为 "[...省略 {N} 字符...]"            │
│                                                                 │
│  预处理后的消息用于记忆提取                                      │
│  原始消息保留在转录文件中不变                                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**消息内容清洗规则**:

| 内容类型 | 处理方式 | 原因 |
|----------|---------|------|
| 代码块 | 替换为 `[代码: {lang}, {lines}行]` | 代码几乎不含可提取的个人记忆 |
| JSON/XML 数据 | 替换为 `[结构化数据: {类型}]` | 同上 |
| 工具调用结果 | 截断到 200 字符 | 仅保留概要 |
| 文件路径列表 | 保留前 5 条 + 计数 | 避免占用 token |
| 说明性文字 | 完整保留 | 最可能含有记忆信号 |

### 5.4 提取触发时机

**参考社区最佳实践** (Mem0, MemGPT, Zep, Amazon Bedrock AgentCore):
- 多条件组合触发：消息计数 + 空闲超时 + 会话结束
- OpenClaw 的 Memory Flush 机制：上下文接近压缩阈值时静默触发
- 闲时处理：定时任务 + 系统负载监控

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        提取触发时机 (修订版 v2)                               │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  ┌───────────────────────────────────────────────────────────────────┐     │
│  │ 触发 1: 每轮对话后 - 显式命令即时提取                              │     │
│  │                                                                    │     │
│  │ 用户消息到达                                                       │     │
│  │     │                                                              │     │
│  │     ├── 追加到转录文件 (JSONL)                                     │     │
│  │     │                                                              │     │
│  │     ├── 正则匹配显式命令 ("记住: xxx" / "remember: xxx")           │     │
│  │     │   └── 命中 → 立即提取，不入队列                              │     │
│  │     │                                                              │     │
│  │     └── 正则匹配隐式信号 → 累加信号计数器                          │     │
│  │                                                                    │     │
│  │ 特点: 同步处理，零延迟                                             │     │
│  └───────────────────────────────────────────────────────────────────┘     │
│                                                                             │
│  ┌───────────────────────────────────────────────────────────────────┐     │
│  │ 触发 2: 段满触发 - 累积够一段后异步提取                            │     │
│  │                                                                    │     │
│  │ 消息累积计数 >= segmentSize (5条)?                                  │     │
│  │     │                                                              │     │
│  │     ├── YES → 构建当前段 (含 overlap 上下文)                       │     │
│  │     │         └── 放入提取队列                                     │     │
│  │     │         └── 重置计数器 (保留 overlap 部分)                    │     │
│  │     │         └── 清除空闲定时器                                   │     │
│  │     │                                                              │     │
│  │     └── NO → 启动/重置空闲定时器                                   │     │
│  │                                                                    │     │
│  │ 特点: 自适应，对话越长提取越多段                                    │     │
│  └───────────────────────────────────────────────────────────────────┘     │
│                                                                             │
│  ┌───────────────────────────────────────────────────────────────────┐     │
│  │ 触发 3: 空闲超时触发 - 解决短对话遗漏问题 (新增)                   │     │
│  │                                                                    │     │
│  │ 用户只发了 3 轮对话就停止，不满 segmentSize (5条)                  │     │
│  │     │                                                              │     │
│  │     ├── 最后一条消息后启动空闲定时器                               │     │
│  │     │                                                              │     │
│  │     ├── 60秒内无新消息 → 触发提取                                  │     │
│  │     │   └── 提取未满一段的累积消息                                 │     │
│  │     │                                                              │     │
│  │     └── 60秒内有新消息 → 重置定时器                                │     │
│  │                                                                    │     │
│  │ 特点: 确保短对话也能被提取，不会因为不满段而遗漏                   │     │
│  │ 参考: Amazon Bedrock AgentCore 的 idle_timeout 触发               │     │
│  └───────────────────────────────────────────────────────────────────┘     │
│                                                                             │
│  ┌───────────────────────────────────────────────────────────────────┐     │
│  │ 触发 4: 会话结束 - 全量分段提取                                    │     │
│  │                                                                    │     │
│  │ 会话结束 (Agent destroy / 超时 / 用户关闭)                         │     │
│  │     │                                                              │     │
│  │     ├── 清除空闲定时器                                             │     │
│  │     │                                                              │     │
│  │     ├── 读取完整转录文件                                           │     │
│  │     │                                                              │     │
│  │     ├── 计算已提取过的段 (通过 extraction_progress 表)             │     │
│  │     │                                                              │     │
│  │     ├── 仅处理未提取的剩余消息                                     │     │
│  │     │   ├── 不足一段 → 作为尾部段处理                              │     │
│  │     │   └── 多于一段 → 继续分段处理                                │     │
│  │     │                                                              │     │
│  │     └── 全部段处理完毕后标记会话提取完成                            │     │
│  │                                                                    │     │
│  │ 特点: 确保无遗漏，回捞所有未处理消息                                │     │
│  └───────────────────────────────────────────────────────────────────┘     │
│                                                                             │
│  ┌───────────────────────────────────────────────────────────────────┐     │
│  │ 触发 5: 新会话开始时处理遗留 (可选)                                │     │
│  │                                                                    │     │
│  │ 新会话开始前                                                       │     │
│  │     │                                                              │     │
│  │     ├── 检查是否有上一次会话未提取的消息                           │     │
│  │     │   └── (通过 extraction_progress 表状态判断)                  │     │
│  │     │                                                              │     │
│  │     └── 如有遗留 → 触发提取                                        │     │
│  │                                                                    │     │
│  │ 特点: 处理应用崩溃等异常情况下的遗漏                               │     │
│  │ 参考: OpenClaw 的断点恢复机制                                      │     │
│  └───────────────────────────────────────────────────────────────────┘     │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 5.5 提取进度追踪

**问题**: 会话中途提取和会话结束提取需要知道哪些消息已经处理过。

**方案**: 在 SQLite 中记录提取进度。

```sql
-- 提取进度表 (新增)
CREATE TABLE IF NOT EXISTS extraction_progress (
  session_id TEXT NOT NULL,           -- 会话 ID
  segment_index INTEGER NOT NULL,     -- 段序号 (0-based)
  start_msg_index INTEGER NOT NULL,   -- 段起始消息索引
  end_msg_index INTEGER NOT NULL,     -- 段结束消息索引 (exclusive)
  status TEXT NOT NULL DEFAULT 'pending',  -- pending/processing/completed/failed
  memories_extracted INTEGER DEFAULT 0,    -- 本段提取的记忆数
  created_at INTEGER NOT NULL,
  completed_at INTEGER,
  error_message TEXT,                 -- 失败原因
  PRIMARY KEY (session_id, segment_index)
);

CREATE INDEX IF NOT EXISTS idx_extraction_progress_session
  ON extraction_progress(session_id, status);
```

**进度追踪流程**:

```
┌─────────────────────────────────────────────────────────────────┐
│                   提取进度管理                                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  段满触发时:                                                     │
│  1. INSERT extraction_progress (session_id, segment_index,      │
│     start_msg_index, end_msg_index, status='pending')           │
│  2. 放入提取队列                                                 │
│  3. 队列处理完成 → UPDATE status='completed'                    │
│                                                                 │
│  会话结束时:                                                     │
│  1. 读取转录文件总消息数                                         │
│  2. SELECT MAX(end_msg_index) FROM extraction_progress          │
│     WHERE session_id = ? AND status = 'completed'               │
│  3. 未处理范围 = [max_end_msg_index, total_messages)            │
│  4. 对未处理范围分段 → 入队列                                    │
│                                                                 │
│  断点恢复 (客户端重启后):                                        │
│  1. SELECT * FROM extraction_progress WHERE status = 'pending'  │
│  2. 对 pending 的段重新入队列                                    │
│  3. 继续处理                                                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 5.6 方案对比

| 方案 | 触发时机 | 提取方式 | API 成本 | 准确率 | 适用场景 |
|------|----------|----------|----------|--------|----------|
| **方案 A: 纯正则** | 每轮对话 | 正则+规则 | 无 | 中等 | 快速、低成本 |
| **方案 B: 纯 LLM** | 每 N 轮 | LLM 提取 | 高 | 高 | 高质量、不敏感成本 |
| **方案 C: 混合方案** | 每轮对话 | 正则筛选+LLM验证 | 中 | 高 | 平衡型 (推荐) |
| **方案 D: 会话结束** | 会话结束 | LLM 批量 | 低 | 高 | 上下文完整 |
| **方案 E: 分段混合 (新增)** | 段满+会话结束 | 分段提取+去重 | 中 | 高 | **大对话场景 (推荐)** |

### 5.7 推荐方案: 分段混合提取策略

### 5.8 跨段去重机制

由于分段有重叠 (overlap)，不同段可能提取出相同或相似的记忆。需要去重:

```
┌─────────────────────────────────────────────────────────────────┐
│                     跨段去重策略                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  新提取的记忆候选                                                │
│       │                                                         │
│       ▼                                                         │
│  Step 1: 精确去重 (fingerprint)                                  │
│  ├── 计算 SHA256(normalize(text))                               │
│  ├── 查询 memories 表: WHERE fingerprint = ?                    │
│  └── 命中 → 直接跳过                                            │
│       │                                                         │
│       ▼                                                         │
│  Step 2: 近似去重 (文本相似度)                                   │
│  ├── 与同一会话最近提取的记忆比较                                │
│  ├── 计算编辑距离 / Jaccard 相似度                               │
│  ├── 相似度 > 0.8 → 保留置信度更高的那条                        │
│  └── 相似度 <= 0.8 → 视为不同记忆                               │
│       │                                                         │
│       ▼                                                         │
│  Step 3: 语义去重 (可选, 需向量化)                               │
│  ├── 向量相似度 > 0.95 → 合并                                   │
│  └── 向量相似度 <= 0.95 → 保留                                  │
│                                                                 │
│  normalize(text):                                                │
│  ├── 转小写                                                     │
│  ├── 移除标点符号                                                │
│  ├── 移除多余空白                                                │
│  └── 排序关键词 (可选)                                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 5.9 LLM 提取 Prompt 设计

#### 5.9.1 分段提取 Prompt (核心)

```
你是一个记忆提取助手。请从以下对话片段中提取值得长期记忆的信息。

## 对话片段 (第 {segment_index}/{total_segments} 段)
{segment_content}

## 已知记忆 (避免重复)
{existing_memories_summary}

## 提取规则
1. 只提取关于用户的个人事实、偏好、习惯、重要决策
2. 每条记忆应该是独立的、完整的陈述
3. 忽略:
   - 临时性信息 (如"我今天很累")
   - 纯粹的问题或指令
   - 与已知记忆重复的信息
   - 代码内容、技术实现细节
   - 工具调用和执行结果
4. 如果对话片段中没有值得记忆的信息，返回空数组

## 输出格式
返回 JSON 数组，每条记忆包含:
{
  "text": "记忆文本 (独立完整的陈述，不依赖上下文即可理解)",
  "category": "fact|preference|event|decision",
  "confidence": 0.0-1.0
}

如果没有值得记忆的信息，返回空数组 []
```

**关键改进**:
- 标注段序号，让 LLM 知道这是部分上下文
- 传入已知记忆摘要，避免提取重复内容
- 新增 `decision` 类别，捕获重要决策
- 要求记忆文本"独立完整"，不依赖上下文

#### 5.9.2 验证 Prompt (不变)

```
你是一个记忆验证助手。请判断以下候选记忆是否值得保存。

## 候选记忆
{candidate_memory}

## 现有记忆
{existing_memories}

## 判断规则
1. 是否与现有记忆冲突或重复?
2. 是否具有长期价值?
3. 是否是用户个人信息而非通用知识?

## 输出
{
  "accept": true/false,
  "reason": "拒绝或接受的原因",
  "merged_text": "如果需要合并，提供合并后的文本",
  "confidence": 0.0-1.0
}
```

### 5.13 与 OpenClaw 记忆提取方案的对比

| 维度 | 本方案 (Nuwax Agent) | OpenClaw |
|------|---------------------|----------|
| **触发时机** | 段满 + 会话结束 + 显式命令 | 仅 /new 或 /reset 命令触发 |
| **消息持久化** | JSONL 转录文件 (实时追加) | JSONL 会话文件 (框架自动记录) |
| **消息窗口** | 分段 (5条/段, 2条 overlap) | 固定取最近 15 条消息 |
| **大消息处理** | 代码块替换 + 截断 + 摘要 | 仅过滤 tool_use，不做截断 |
| **LLM 提取** | 分段 Prompt，每段独立提取 | 仅生成 1-2 词 slug，不做内容提取 |
| **去重机制** | 三层去重 (hash + 文本相似度 + 向量) | 仅文件级 hash 去重 |
| **断点恢复** | extraction_progress 表追踪进度 | 基于文件 mtime/hash 增量同步 |
| **内容索引** | 文件同步到 SQLite → FTS5 + 向量 | 直接分块 → 嵌入 → SQLite |
| **记忆层次** | 每日记忆 + MEMORY.md 核心记忆 | 平铺 Markdown 文件 |
| **提炼机制** | 定时 LLM 合并到 MEMORY.md | 无自动提炼 |

**借鉴 OpenClaw 的关键点**:

1. **会话转录持久化**: OpenClaw 的 JSONL 会话文件是记忆提取的基础。本方案新增 `transcripts/` 目录，采用相同的 JSONL 格式
2. **分块策略参数**: OpenClaw 使用 256 token/chunk, 32 token overlap。本方案在消息级分段 (5条/段, 2条 overlap)，粒度更粗但更适合对话场景
3. **批量嵌入控制**: OpenClaw 按 8000 token/batch、4 并发控制 API 调用。本方案在嵌入环节可直接复用此策略
4. **脱敏处理**: OpenClaw 的 `redactSensitiveText` 在索引前移除敏感信息。本方案在转录写入时即做脱敏
5. **文件监听同步**: OpenClaw 的 mtime + hash 变更检测机制，与本方案的文件同步设计一致

**本方案的差异化优势**:

1. **实时提取**: 不等会话结束，段满即提取，减少信息遗漏
2. **语义提取**: 使用 LLM 提取结构化记忆，而非简单记录会话摘要
3. **分层记忆**: 每日记忆 + 核心记忆的双层结构，支持定时提炼
4. **断点恢复**: 进程崩溃后可从 extraction_progress 恢复，不丢失进度

```
│  │ 清理 2: 会话转录文件 (.memory/transcripts/*.jsonl)         │ │
│  │                                                            │ │
│  │ 保留天数: transcriptRetentionDays (默认 7 天)              │ │
│  │                                                            │ │
│  │ 流程:                                                      │ │
│  │ ├── cutoff = 当前时间 - 保留天数                           │ │
│  │ ├── 扫描 .memory/transcripts/ 目录                         │ │
│  │ ├── 对每个 .jsonl 文件:                                    │ │
│  │ │   ├── 文件 mtime < cutoff → 删除文件                     │ │
│  │ │   └── 文件 mtime >= cutoff → 保留                        │ │
│  │ └── 记录清理统计                                            │ │
│  │                                                            │ │
│  │ 注意:                                                      │ │
│  │ ├── 使用 mtime 而非文件名判断 (文件名是 sessionId)         │ │
│  │ └── 正在进行中的会话 (agentService active) 的转录不清理    │ │
│  └───────────────────────────────────────────────────────────┘ │
│                                                                 │
│  ┌───────────────────────────────────────────────────────────┐ │
│  │ 清理 3: 提取进度记录 (extraction_progress 表)              │ │
│  │                                                            │ │
│  │ 保留天数: transcriptRetentionDays (默认 7 天，与转录一致)  │ │
│  │                                                            │ │
│  │ 流程:                                                      │ │
│  │ ├── cutoff = 当前时间 - 保留天数                           │ │
│  │ ├── 删除已完成的过期记录:                                   │ │
│  │ │   DELETE FROM extraction_progress                         │ │
│  │ │   WHERE status = 'completed'                             │ │
│  │ │     AND completed_at < cutoff                            │ │
│  │ │                                                          │ │
│  │ ├── 删除失败的过期记录:                                     │ │
│  │ │   DELETE FROM extraction_progress                         │ │
│  │ │   WHERE status = 'failed'                                │ │
│  │ │     AND created_at < cutoff                              │ │
│  │ │                                                          │ │
│  │ └── 不删除 pending/processing 状态的记录                    │ │
│  │     (可能是断点恢复需要的)                                  │ │
│  └───────────────────────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**清理策略汇总**:

| 清理对象 | 保留时间 | 判断依据 | 保护条件 |
|----------|---------|---------|---------|
| 每日记忆文件 | 30 天 | 文件名中的日期 | 无 |
| 转录文件 | 7 天 | 文件 mtime | 活跃会话不清理 |
| 进度记录 (completed) | 7 天 | completed_at | 无 |
| 进度记录 (failed) | 7 天 | created_at | 无 |
| 进度记录 (pending/processing) | 不清理 | — | 断点恢复需要 |

**异常处理**:

| 场景 | 处理 |
|------|------|
| 转录已删除但进度记录还在 pending | 标记 status='failed', error='transcript_expired' |
| 进度记录 pending 超过 24 小时 | 视为僵尸记录，标记 status='failed', error='stale' |
| 清理过程中出错 | 记录日志，跳过该文件，继续处理下一个 |

## 6. 数据库 Schema 设计

> 参考 openclaw 的设计模式，采用混合索引策略：向量搜索 (sqlite-vec) + 全文搜索 (FTS5)。

### 6.1 核心表结构

#### 6.1.1 memories 表 (记忆主表)

对应 openclaw 的 `chunks` 表，存储所有记忆片段。

```sql
-- 记忆主表 (对应 openclaw 的 chunks 表)
CREATE TABLE IF NOT EXISTS memories (
  -- 主键与内容
  id TEXT PRIMARY KEY,              -- 主键 (基于内容 Hash 生成)
  text TEXT NOT NULL,               -- 记忆文本内容
  fingerprint TEXT NOT NULL,        -- SHA256 去重指纹

  -- 分类与权重
  category TEXT NOT NULL DEFAULT 'fact',    -- 分类: fact/preference/event/skill
  confidence REAL NOT NULL DEFAULT 0.75,    -- 置信度 (0-1)
  is_explicit INTEGER NOT NULL DEFAULT 0,   -- 是否显式记忆 (0/1)
  importance REAL NOT NULL DEFAULT 0.5,     -- 重要性权重 (0-1)

  -- 来源追踪 (关键: 支持单向同步)
  source TEXT NOT NULL,             -- 来源: core/daily
  source_path TEXT NOT NULL,        -- 来源文件路径 (如 MEMORY.md, memory/2025-03-02.md)
  start_line INTEGER,               -- Markdown 起始行号 (1-indexed)
  end_line INTEGER,                 -- Markdown 结束行号 (1-indexed)

  -- 向量相关 (参考 openclaw chunks 表)
  embedding BLOB,                   -- 向量数据 (Float32Array 序列化)
  embedding_model TEXT,             -- 使用的向量模型
  embedding_dims INTEGER,           -- 向量维度

  -- 生命周期
  status TEXT NOT NULL DEFAULT 'active',   -- 状态: active/archived/deleted
  access_count INTEGER NOT NULL DEFAULT 0, -- 访问次数
  last_accessed_at INTEGER,                -- 最后访问时间戳 (Unix ms)
  created_at INTEGER NOT NULL,             -- 创建时间戳 (Unix ms)
  updated_at INTEGER NOT NULL              -- 更新时间戳 (Unix ms)
);

-- 索引设计
CREATE INDEX IF NOT EXISTS idx_memories_source
  ON memories(source, source_path);
CREATE INDEX IF NOT EXISTS idx_memories_status
  ON memories(status);
CREATE INDEX IF NOT EXISTS idx_memories_fingerprint
  ON memories(fingerprint);
CREATE INDEX IF NOT EXISTS idx_memories_created
  ON memories(created_at DESC);
```

#### 6.1.2 memory_fts 表 (FTS5 全文索引)

```sql
-- FTS5 全文索引虚拟表
-- 使用 unicode61 分词器支持中文
CREATE VIRTUAL TABLE IF NOT EXISTS memory_fts USING fts5(
  text,
  created_at,                    -- 创建时间戳 (方便查看数据)
  content='memories',
  content_rowid='rowid',
  tokenize='unicode61'
);

-- FTS5 同步触发器
-- 注意: 使用外部内容表模式，需要手动维护索引

-- 插入时同步到 FTS
CREATE TRIGGER IF NOT EXISTS memory_ai AFTER INSERT ON memories
WHEN NEW.status = 'active' BEGIN
  INSERT INTO memory_fts(rowid, text, created_at)
  VALUES (NEW.rowid, NEW.text, NEW.created_at);
END;

-- 删除时从 FTS 移除
CREATE TRIGGER IF NOT EXISTS memory_ad AFTER DELETE ON memories BEGIN
  INSERT INTO memory_fts(memory_fts, rowid, text, created_at)
  VALUES ('delete', OLD.rowid, OLD.text, OLD.created_at);
END;

-- 更新时同步 FTS
CREATE TRIGGER IF NOT EXISTS memory_au AFTER UPDATE ON memories
WHEN NEW.status = 'active' BEGIN
  INSERT INTO memory_fts(memory_fts, rowid, text, created_at)
  VALUES ('delete', OLD.rowid, OLD.text, OLD.created_at);
  INSERT INTO memory_fts(rowid, text, created_at)
  VALUES (NEW.rowid, NEW.text, NEW.created_at);
END;

-- 状态变为非活跃时从 FTS 移除
CREATE TRIGGER IF NOT EXISTS memory_archive AFTER UPDATE OF status ON memories
WHEN OLD.status = 'active' AND NEW.status != 'active' BEGIN
  INSERT INTO memory_fts(memory_fts, rowid, text, created_at)
  VALUES ('delete', OLD.rowid, OLD.text, OLD.created_at);
END;
```

#### 6.1.3 memory_vec 表 (向量索引)

```sql
-- sqlite-vec 向量索引虚拟表
-- 维度 1536 对应 OpenAI text-embedding-3-small
-- 实际维度根据配置动态创建
CREATE VIRTUAL TABLE IF NOT EXISTS memory_vec USING vec0(
  memory_id TEXT PRIMARY KEY,
  embedding FLOAT[1536]  -- 维度根据 embedding 模型动态调整
);

-- 向量索引同步触发器 (sqlite-vec 可用时)
-- 插入记忆时同步向量
CREATE TRIGGER IF NOT EXISTS memory_vec_ai AFTER INSERT ON memories
WHEN NEW.embedding IS NOT NULL AND NEW.status = 'active' BEGIN
  INSERT INTO memory_vec(memory_id, embedding)
  VALUES (NEW.id, NEW.embedding);
END;

-- 删除记忆时移除向量
CREATE TRIGGER IF NOT EXISTS memory_vec_ad AFTER DELETE ON memories BEGIN
  DELETE FROM memory_vec WHERE memory_id = OLD.id;
END;

-- 更新记忆时同步向量
CREATE TRIGGER IF NOT EXISTS memory_vec_au AFTER UPDATE OF embedding ON memories
WHEN NEW.embedding IS NOT NULL AND NEW.status = 'active' BEGIN
  DELETE FROM memory_vec WHERE memory_id = OLD.id;
  INSERT INTO memory_vec(memory_id, embedding)
  VALUES (NEW.id, NEW.embedding);
END;

-- 状态变为非活跃时移除向量
CREATE TRIGGER IF NOT EXISTS memory_vec_archive AFTER UPDATE OF status ON memories
WHEN OLD.status = 'active' AND NEW.status != 'active' BEGIN
  DELETE FROM memory_vec WHERE memory_id = OLD.id;
END;
```

## 7. 检索策略

### 7.1 搜索范围

```
┌─────────────────────────────────────────────────────────────────┐
│                     搜索范围                                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  始终包含:                                                       │
│  └── MEMORY.md (核心长期记忆)                                   │
│                                                                 │
│  动态包含 (可配置最近 N 天):                                     │
│  ├── memory/2025-03-02.md (今天)                               │
│  └── memory/2025-03-01.md (昨天)                               │
│                                                                 │
│  默认配置: N = 2 天                                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 7.2 三层降级检索逻辑

```
┌─────────────────────────────────────────────────────────────────┐
│                   三层降级检索                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  用户查询 (query)                                               │
│       │                                                         │
│       ▼                                                         │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ Layer 1: FTS5 全文搜索 (始终执行)                        │   │
│  │                                                          │   │
│  │ 流程:                                                    │   │
│  │ 1. 清理查询字符串 (转义特殊字符)                         │   │
│  │ 2. 执行 FTS5 MATCH 查询                                  │   │
│  │ 3. 使用 BM25 排序                                        │   │
│  │ 4. 返回 top-K 结果                                       │   │
│  └─────────────────────────────────────────────────────────┘   │
│       │                                                         │
│       ▼                                                         │
│  Embedding 已配置?                                              │
│       │                                                         │
│       ├── NO → 跳过向量搜索，直接返回 FTS 结果                 │
│       │                                                         │
│       └── YES → 获取查询向量                                   │
│              │                                                  │
│              ▼                                                  │
│       ┌─────────────────────────────────────────────────────┐  │
│       │ Layer 2: 向量搜索                                    │  │
│       │                                                      │  │
│       │ 尝试 sqlite-vec 扩展                                 │  │
│       │     │                                                │  │
│       │     ├── 可用 → 使用原生 vec_distance_cosine         │  │
│       │     │                                                │  │
│       │     └── 不可用 → 降级到 JS 余弦相似度计算            │  │
│       │                                                      │  │
│       │ 返回 top-K 结果                                      │  │
│       └─────────────────────────────────────────────────────┘  │
│              │                                                  │
│              ▼                                                  │
│       混合合并 (Hybrid Merge)                                   │
│              │                                                  │
│              ├── 归一化各来源分数                              │
│              ├── 加权合并: finalScore = w_vec * vec + w_fts * fts │
│              └── 排序并返回 top-N                               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

| ftsWeight | 0.3 | FTS 检索权重 |
| limit | 12 | 返回结果数量 |
| minScore | 0.4 | 最低分数阈值 |

## 8. 向量化机制

### 8.1 向量状态管理

**三态设计**:

| 状态 | 值 | 含义 |
|------|-----|------|
| 未检测 | null | 尚未尝试加载 sqlite-vec |
| 可用 | true | sqlite-vec 加载成功 |
| 不可用 | false | sqlite-vec 加载失败 |

**检测流程**:

```
┌─────────────────────────────────────────────────────────────────┐
│                 sqlite-vec 可用性检测                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 检查配置是否启用向量化                                       │
│     └── 未启用 → 直接返回 false                                 │
│                                                                 │
│  2. 检查是否已检测过 (缓存结果)                                  │
│     └── 已检测 → 返回缓存结果                                   │
│                                                                 │
│  3. 尝试加载 sqlite-vec 扩展                                     │
│     ├── 导入 sqlite-vec 包                                      │
│     ├── 获取扩展路径                                            │
│     ├── 启用扩展加载                                            │
│     └── 加载扩展                                                │
│                                                                 │
│  4. 记录结果                                                     │
│     ├── 成功 → available = true, 记录扩展路径                   │
│     └── 失败 → available = false, 记录错误信息                  │
│                                                                 │
│  5. 超时保护 (30秒)                                              │
│     └── 超时 → 视为失败                                         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 8.2 模型切换处理

**触发条件**: 用户更改 Embedding 模型配置时

**处理流程**:

```
┌─────────────────────────────────────────────────────────────────┐
│                   模型切换处理                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 检测模型变更                                                 │
│     ├── 读取数据库中的 model 和 dims 元数据                     │
│     └── 与新配置比对                                            │
│                                                                 │
│  2. 无变更则跳过                                                 │
│                                                                 │
│  3. 创建临时数据库                                               │
│     └── 复制 Schema 结构                                        │
│                                                                 │
│  4. 迁移非向量数据                                               │
│     └── 复制所有记忆记录 (不含 embedding 字段)                  │
│                                                                 │
│  5. 重新向量化所有记忆                                           │
│     ├── 遍历所有记忆文本                                        │
│     ├── 调用新模型 API 获取向量                                 │
│     └── 写入临时数据库                                          │
│                                                                 │
│  6. 重建向量索引                                                 │
│     └── 使用新维度创建 vec0 虚拟表                              │
│                                                                 │
│  7. 原子性交换                                                   │
│     ├── 备份原数据库                                            │
│     ├── 临时数据库替换原数据库                                  │
│     └── 删除备份                                                │
│                                                                 │
│  8. 更新元数据                                                   │
│     └── 写入新的 model 和 dims                                  │
│                                                                 │
│  9. 错误恢复                                                     │
│     └── 任何步骤失败 → 回滚到原数据库                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 8.3 向量缓存策略

**目的**: 避免重复计算相同内容的向量

**缓存逻辑**:

```
┌─────────────────────────────────────────────────────────────────┐
│                   向量缓存逻辑                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  获取向量时:                                                     │
│                                                                 │
│  1. 计算内容 Hash                                                │
│     └── hash = SHA256(文本内容)                                 │
│                                                                 │
│  2. 查询缓存表                                                   │
│     └── SELECT embedding FROM embedding_cache                   │
│         WHERE content_hash = ? AND model = ?                    │
│                                                                 │
│  3. 缓存命中?                                                    │
│     ├── YES → 返回缓存向量，更新访问时间和计数                  │
│     └── NO → 调用 API 获取向量                                  │
│                                                                 │
│  4. 新向量写入缓存                                               │
│     └── INSERT INTO embedding_cache (...)                       │
│                                                                 │
│  5. LRU 淘汰 (可选)                                              │
│     ├── 检查缓存数量是否超过上限                                │
│     └── 超过则删除最久未访问的记录                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 9. Markdown 分块逻辑

### 9.1 分块策略

**目的**: 将长 Markdown 文件拆分为适合向量化的片段

**分块规则**:

```
┌─────────────────────────────────────────────────────────────────┐
│                   分块规则                                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 按行遍历文件内容                                             │
│                                                                 │
│  2. 检测到 ## 标题时                                             │
│     ├── 保存当前 chunk (如果有内容)                             │
│     └── 开始新的 chunk                                          │
│                                                                 │
│  3. chunk 超过最大字符数时                                       │
│     ├── 保存当前 chunk                                          │
│     ├── 保留 overlap 字符 (与下一 chunk 重叠)                   │
│     └── 开始新的 chunk                                          │
│                                                                 │
│  4. 文件结束时                                                   │
│     └── 保存最后一个 chunk                                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**配置参数**:

| 参数 | 默认值 | 说明 |
|------|--------|------|
| maxChars | 1000 | 单个 chunk 最大字符数 |
| overlapChars | 100 | 相邻 chunk 重叠字符数 |

### 9.2 Chunk 数据结构

每个 chunk 记录以下信息:

| 字段 | 说明 |
|------|------|
| startLine | 起始行号 (1-indexed) |
| endLine | 结束行号 (1-indexed) |
| text | chunk 文本内容 |
| hash | SHA256 哈希值 (用于变更检测) |

## 15. 总结

### 15.1 方案特点

| 特点 | 说明 |
|------|------|
| **文件为主** | Markdown 是 Source of Truth，人类可编辑 |
| **单向同步** | Markdown → SQLite，简单可靠 |
| **三层降级** | sqlite-vec → JS 向量 → 关键词，确保可用 |
| **双重触发** | 文件监控 + Hash 校验，确保同步及时 |
| **定时提炼** | 每日 00:00 合并最近 2 天记忆 |
| **自动清理** | 超过保留天数的每日文件自动删除 |

### 15.2 与其他方案对比

| 维度 | 本方案 | Mem0 | Letta |
|------|--------|------|-------|
| 人类可编辑 | ✅ Markdown | ❌ | ❌ |
| 向量依赖 | 可选 | 必需 | 必需 |
| Agent 侵入 | 无 | 低 | 高 |
| 自动提取 | 有 | 有 | 无 |
| 部署成本 | 低 | 中 | 高 |
| 定时提炼 | 有 | 无 | 无 |

### 15.3 关键成功因素

1. **先做对，再做快** - MVP 确保核心功能可靠
2. **保持简单** - 单向同步比双向简单
3. **双重保障** - 文件监控 + 启动校验确保数据一致
4. **渐进增强** - 向量化可选，零配置可用
5. **迭代优化** - 根据使用反馈调整

## 16. API 接口约定

### 16.1 ComputerChatRequest 记忆相关字段

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `prompt` | string | 是 | 完整提示词（可能包含系统提示） |
| `original_user_prompt` | string | 否 | 原始用户提示词（纯净用户输入） |
| `open_long_memory` | boolean | 否 | 是否开启长期记忆（默认 false） |
| `system_prompt` | string | 否 | 系统提示词（通过 ACP _meta 传递） |

### 16.2 字段使用逻辑

```
记忆提取 → 仅使用 original_user_prompt，为空时打印错误日志并跳过
记忆搜索 → 仅使用 original_user_prompt，为空时打印错误日志并跳过
记忆注入 → 根据 open_long_memory 决定是否注入（默认 false）
```

### 16.3 字段要求

- `original_user_prompt` **必须**由客户端提供纯净用户输入
- 如果 `original_user_prompt` 为空，打印错误日志，不回退到 `prompt`
- 如果 `open_long_memory` 未提供，默认为 false（禁用记忆）

### 16.4 请求示例

**开启记忆（需显式设置）**:
```json
{
  "prompt": "混合内容（系统提示+用户消息）...",
  "original_user_prompt": "我喜欢看NBA比赛",
  "open_long_memory": true
}
```
预期: 记忆正常提取和注入

**关闭记忆（默认行为）**:
```json
{
  "prompt": "混合内容（系统提示+用户消息）...",
  "original_user_prompt": "我喜欢看NBA比赛",
  "open_long_memory": false
}
```
预期: 不注入记忆内容，不记录用户消息

**original_user_prompt 为空（错误情况）**:
```json
{
  "prompt": "混合内容...",
  "original_user_prompt": ""
```
